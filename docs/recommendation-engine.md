# Рекомендательная логика

Статус: обновлено 2026-06-17 с учетом доработок за 2026-06-15 - 2026-06-17.

Документ объясняет, как бот подбирает программы, какие данные учитывает и почему рекомендации могут отличаться между сценариями.

## Где живет логика

| Задача | Файл |
| --- | --- |
| Общий подбор программ | `src/recommendations.js` |
| Быстрый сценарий по описанию | `src/description-flow.js`, `src/description-selection.js` |
| Пошаговый сценарий | `src/index.js`, состояния `s2_*` |
| Углубленная траектория | `src/deep-trajectory.js` |
| Траектория новых интересов | `src/deep-trajectory.js`, режим `wide` |
| Единая скоринговая модель | `src/scoring-model.js` |
| Онтология запроса | `src/query-ontology.js` |
| Чтение PFDO-зеркала | `src/pfdo-mirror.js` |
| История рекомендаций | `src/session-store.js` |

## Общий принцип

Бот не выбирает программу одним правилом. Он строит профиль запроса, получает кандидатов из локального PFDO-зеркала, применяет общие фильтры, начисляет баллы в `src/scoring-model.js` и затем нормализует результат в пользовательскую карточку.

Профиль обычно включает:

- возраст;
- интересы;
- цель обучения;
- населенный пункт или организацию;
- бюджет;
- удобное расписание;
- формат занятий;
- ограничения и нежелательные форматы;
- признаки адаптации ребенка к новому.

Одна модель скоринга обслуживает четыре сценария:

- `s1` — подбор по свободному описанию;
- `s2` — пошаговая анкета;
- `s3` — углубление по уже пройденным темам;
- `s4` — новые интересы без повторения уже изученных категорий.

Согласованные критерии и веса описаны отдельно в [Скоринговой модели](Скоринговая%20модель.md).

## Источник каталога

Основной источник - база `pfdo_51_mirror`. Перед подбором код проверяет, что в таблице `pfdo_programs` есть записи.

Если зеркало доступно, используются функции:

- `getMunicipalities()` - список муниципалитетов;
- `searchPrograms()` - список программ-кандидатов;
- `getProgramDetail()` - подробный payload программы.

Если зеркало недоступно:

- быстрый сценарий в строгом режиме возвращает пустой результат;
- нестрогий вызов может перейти к встроенному mock-каталогу.

Mock-каталог нужен для разработки и демонстраций. Его нельзя считать продуктивным источником.

## Confidence

`confidence` показывает не качество программы, а полноту пользовательского профиля.

| Уровень | Когда появляется |
| --- | --- |
| `high` | Достаточно интересов, ограничений, цели и признаков адаптации. |
| `medium` | Есть часть важных параметров, но профиль не полный. |
| `low` | Данных мало или они слишком неопределенные. |

На `confidence` влияют:

- количество интересов;
- наличие ограничений;
- цель обучения;
- сигналы адаптации;
- слишком широкий или неопределенный запрос.

## Фильтры до ранжирования

Перед начислением баллов программа может быть исключена.

Жесткие отсечки:

- возраст ребенка не входит в возрастной диапазон программы;
- возраст ребенка не входит в рекомендованный диапазон конкретной группы;
- программа не получила тематическую связь в углубленном сценарии.

Если пользователь указал населенный пункт, бот сначала пытается сопоставить его со справочником муниципалитетов. Сопоставление нормализует слова вроде `город`, `район`, `округ`, `область`.

## Скоринг общего подбора

Для программ из PFDO-зеркала скоринг учитывает текст:

- название программы;
- направление;
- организацию;
- ключевые слова;
- адрес;
- данные открытых групп.

### Интересы и точные активности

Интересы превращаются в внутренние категории:

| Категория | Примеры совпадений |
| --- | --- |
| `creative` | рисование, дизайн, театр, музыка, танцы, творчество |
| `building` | робототехника, конструирование, инженерия, моделирование, 3D |
| `sports` | спорт, плавание, футбол, хоккей, борьба, гимнастика, туризм |
| `social` | театр, журналистика, лидерство, дебаты, коммуникация |
| `logic` | математика, программирование, физика, информатика, логика |
| `calm` | мастерская, студия, творчество, шахматы, моделирование |

Совпадение интереса с программой добавляет баллы. Если интересы указаны, но совпадений нет, программа получает штраф или отсекается как `interest_mismatch`.

### Точный интерес из запроса

Если пользователь называет конкретную активность, бот дополнительно выделяет `specificInterestTerms`. Например, запрос `занятия по баскетболу` дает широкий интерес `sports` и точные термины `баскетбол`, `баскет`.

Точные термины сопоставляются с программой в таком порядке:

1. уровень 3 классификатора тем: нормализованная тема программы;
2. уровень 2 классификатора тем: категория темы;
3. уровень 1 классификатора тем: родительская категория;
4. сильный fallback по названию, поисковому названию и ключевым словам;
5. слабый fallback по аннотации, задаче и краткому описанию, если тем у программы нет.

Текущий классификатор тем используется как онтология на стороне программ. Небольшая онтология запроса в `src/query-ontology.js` связывает пользовательские слова и синонимы с широкими интересами и точными терминами. Если пользователь пишет неизвестную активность после явного маркера вроде `по`, `любит`, `ищем`, бот сохраняет ее как сырой точный термин, но не пытается уверенно расширять его до широкой категории без явного правила.

Для запросов с точным интересом выдача сортируется не только по баллам, но и по продуктовым bucket-ам:

1. точная активность, возраст подходит, есть свободные места;
2. точная активность, возраст подходит, но свободных мест нет или набор закрыт;
3. точная активность, но возраст не подходит;
4. остальные программы широкого направления, например другие спортивные программы.

Отрицания тоже учитываются: запрос вроде `спорт, но не футбол` исключает программы, которые совпали с точным запрещенным термином.

Возраст остается жестким фильтром для широких fallback-программ. Для точного интереса модель может пропустить возрастное несовпадение в отдельный нижний bucket, чтобы бот честно показал, что конкретная программа существует, но возраст не подходит.

### Ограничения

Ограничения уменьшают балл, если программа похожа на нежелательный формат.

Примеры:

- `noise` штрафует спорт, театр, ансамбли и похожие шумные форматы;
- `strict` штрафует кадетские и военные форматы;
- `stage` штрафует сценические активности;
- `routine` штрафует слишком традиционные форматы;
- `intense` штрафует интенсивы, соревнования и спорт.

### Цели

Цели добавляют баллы, если текст программы совпадает с запросом:

- первый опыт;
- интерес к теме;
- развитие сильных сторон;
- общение;
- дисциплина;
- проба нового направления.

Суммарный вклад целей ограничен, чтобы одна группа признаков не подавляла все остальные.

### Направленность

Если пользователь выбрал конкретную направленность, совпадение с текстом программы дает сильный плюс. Значение `Не важно` не влияет на баллы.

### Адаптация и формат группы

Бот учитывает сигналы:

- мягкий вход;
- осторожный ребенок;
- быстрый темп;
- спокойная малая группа;
- активная группа;
- структурированная программа;
- свободный творческий формат;
- практическая работа руками;
- логика;
- социальность.

Эти признаки помогают отличать похожие программы по тому, насколько они подходят ребенку по способу включения в занятия.

## Скоринг групп

После ранжирования программ бот загружает detail payload и выбирает наиболее подходящую открытую группу.

Группа получает баллы за:

- цену в пределах бюджета;
- совпадение дней недели;
- вечернее время, если оно запрошено;
- будни или выходные;
- возрастной диапазон;
- наличие свободных мест.

Если нет открытых групп или расписания, карточка все равно может попасть в результат, но бот пишет, что детали нужно проверить на карточке PFDO.

## Нормализация карточки результата

После скоринга программа превращается в пользовательскую карточку:

- `program` - название;
- `formatLabel` - короткая метка;
- `venue` - организация;
- `district` - адрес или локация;
- `schedule` - краткое расписание;
- `price` - цена или рекомендация смотреть условия на карточке;
- `summary` - объяснение, почему программа подходит;
- `note` - что важно проверить;
- `sourceUrl` - ссылка на карточку PFDO.

## Метки формата

Метка выбирается эвристически:

| Метка | Когда используется |
| --- | --- |
| `Лучший вариант для старта` | Есть открытые группы и пользователь ищет первый опыт или мягкий вход. |
| `Подойдет, если важнее общение` | Цель профиля - общение. |
| `Хороший вариант на вырост` | Направление похоже на техническое, инженерное или математическое. |
| `Можно попробовать сейчас` | Есть открытые группы, но нет специальных признаков выше. |
| `Нужно уточнить условия` | Не хватает данных по группам. |

## Пустой результат

Пустой результат может означать разные ситуации:

- зеркало PFDO недоступно;
- в выбранном муниципалитете нет кандидатов;
- кандидаты есть, но они отфильтрованы возрастом или профилем;
- в строгом режиме запрещен fallback на mock-каталог.

В быстром сценарии пустой результат должен вести к изменению критериев или повторному запуску, а не к демонстрационным данным.

## Углубленная траектория и новые интересы

Углубленная траектория использует отдельную логику из `src/deep-trajectory.js`.

Вход:

- 1-5 ссылок на программы `51.pfdo.ru`;
- населенный пункт;
- возраст ребенка.

Общий поток сценариев 3 и 4:

1. Извлекает program id из ссылок.
2. Загружает найденные программы из `pfdo_51_mirror`; при отсутствии карточки может запустить on-demand импорт.
3. Загружает темы из `pfdo_program_topic_aggregates`.
4. Подтягивает классификации из `pfdo_program_topic_classifications`.
5. Показывает обзор пройденных программ и тем.
6. Собирает или уточняет населенный пункт, возраст и критерии поиска.
7. Исключает уже пройденные программы.
8. Ищет кандидатов того же муниципалитета и возраста.
9. Ранжирует кандидатов в режиме `deep` или `wide`.

## Профиль пройденных тем

Профиль включает:

- ключи нормализованных тем;
- названия тем;
- категории тем;
- направления программ;
- суммарные часы;
- среднее количество часов на тему.

Если темы не распознаны, бот может использовать fallback по тексту программы и направлению, но качество такого подбора ниже.

## Скоринг углубления

Кандидат получает баллы за:

- совпадение нормализованных тем;
- совпадение категории в том же направлении;
- новые темы из уже знакомой категории;
- совпадение направления;
- fallback-совпадение по тексту;
- признаки более продвинутого уровня.

Признаки углубления:

- слова про проекты, исследования, соревнования, олимпиады;
- программирование, инженерия, конструирование и моделирование;
- возрастной диапазон, рассчитанный на текущий или более старший возраст;
- больший объем часов по связанным темам;
- заметная практическая часть.

Если программа только повторяет уже пройденные темы и не дает признаков углубления, она получает штраф.

## Скоринг новых интересов

Траектория новых интересов использует тот же профиль пройденных программ, но другой траекторный блок.

Кандидат исключается, если он повторяет уже изученную категорию level 2. Это защищает сценарий от выдачи программ, которые формально подходят по направлению, но по сути повторяют прошлый опыт.

Кандидат получает баллы за:

- новую категорию level 2 в новом разделе level 1;
- новую категорию level 2 в уже знакомом разделе level 1;
- количество содержательных тем;
- уровень программы, подходящий для расширения интересов.

В пользовательском объяснении бот пишет, какие темы новые и чем программа отличается от уже пройденных.

## Что логируется

Каждый результат сохраняется в `recommendation_history`:

- канал;
- id чата или клиента;
- transport metadata пользователя (`userId`, `username`, если доступны);
- источник;
- `confidence`;
- полный payload результата;
- ответы или состояние сценария.

Эта история нужна для анализа качества, отладки и воспроизведения пользовательских кейсов.

## Как проверять изменения

Минимальная проверка:

```bash
npm test
```

Особенно важны тесты:

- `test/recommendations-strict.test.js`;
- `test/description-selection.test.js`;
- `test/description-flow.test.js`;
- `test/scoring-model.test.js`;
- `test/query-ontology.test.js`;
- `test/deep-trajectory.test.js`;
- `test/topic-classifier.test.js`;
- `test/calendar-topics*.test.js`.

После изменения скоринга стоит вручную проверить:

1. Полный запрос с возрастом, местом и интересами.
2. Запрос без одного обязательного поля.
3. Слишком широкий запрос.
4. Запрос с бюджетом и вечерним расписанием.
5. Запрос с конкретным интересом, например `баскетбол`, чтобы точные совпадения шли выше широкого спорта.
6. Углубленную траекторию по нескольким PFDO-ссылкам.
7. Траекторию новых интересов по тем же ссылкам.
8. Случай, когда зеркало PFDO пустое или недоступно.

## Ограничения текущего подхода

- Скоринг основан на правилах и регулярных выражениях, а не на обученной модели ранжирования.
- Качество зависит от полноты `search_name`, `keywords`, `direction`, `address` и detail payload.
- Цена и расписание могут быть неполными в PFDO.
- Углубленная траектория зависит от качества извлечения тем из документов.
- Траектория новых интересов зависит от качества классификатора тем и может отсечь слишком много программ при бедной классификации.
- Веса скоринга не калиброваны на большом наборе ручных оценок.
- Mock-каталог должен использоваться только как fallback для разработки.
