# QA и приемка

Статус: обновлено 2026-06-17 с учетом доработок за 2026-06-15 - 2026-06-17.

Документ описывает, как проверять проект перед релизом, после обновления данных и после изменения сценариев.

## Базовая команда

```bash
npm test
```

Скрипт запускает Node.js test runner по файлам `test/*.test.js`.

## Что покрывают автотесты

| Область | Файлы |
| --- | --- |
| Быстрый сценарий | `test/description-flow.test.js`, `test/description-selection.test.js` |
| Меню и тексты сценариев | `test/flow.test.js`, `test/telegram-menu.test.js` |
| Рекомендации, скоринг и strict fallback | `test/recommendations-strict.test.js`, `test/scoring-model.test.js`, `test/query-ontology.test.js` |

| MAX | `test/max-update.test.js` |

| Классификатор и аналитика тем | `test/topic-classifier.test.js`, `test/topic-analytics-pipeline.test.js` |

## Приемочный минимум перед релизом

Перед релизом должны пройти:

1. `npm test`.
2. Smoke-проверка `/health`.
3. Ручной `/start` в основном пользовательском канале.
4. Минимум один успешный подбор по описанию.
5. Минимум один успешный пошаговый подбор.
6. Проверка PDF-выгрузки.
7. Проверка пустого результата или недоступного PFDO-зеркала, если менялась логика fallback.
8. Проверка углубленной траектории, если менялись темы, PFDO pipeline или `deep-trajectory`.
9. Проверка траектории новых интересов, если менялись темы, PFDO pipeline, `scoring-model` или `deep-trajectory`.
10. Проверка командного меню Telegram и MAX, если менялись `src/telegram-menu.js`, `src/index.js` или тексты сценариев.

## Smoke-проверка сервиса

```bash
curl -sS https://bot.traektoria51.ru/health
```

Ожидается:

- `ok: true`;
- включенные транспорты отображаются корректно;
- сервис отвечает без 5xx.

Локально на сервере:

```bash
curl -sS http://127.0.0.1:3000/health
```

## Ручные сценарии

### Сценарий 1. Подобрать по описанию

Проверка полного запроса:

```text
8 лет, Мурманск, робототехника, лучше вечером, бюджет до 5000
```

Ожидается:

- бот не задает лишние обязательные вопросы;
- показывает резюме или сразу ищет программы;
- результат содержит программы, адрес, расписание, стоимость и ссылку PFDO;
- PDF-кнопка появляется после результата.

Проверка неполного запроса:

```text
хочу кружок по рисованию
```

Ожидается:

- бот просит уточнить возраст и место;
- после уточнения продолжает тот же сценарий;
- не теряет уже введенный интерес.

Проверка редактирования:

```text
не робототехника, а рисование
```

Ожидается:

- бот заменяет интерес;
- возраст и место сохраняются;
- итоговый подбор соответствует исправленному запросу.

### Сценарий 2. Подобрать с AI агентом

Проверьте:

- возраст числом;
- интересы свободным текстом;
- мультивыбор целей;
- расписание;
- формат;
- место;
- стоимость;
- переход к уточнению или результату;
- PDF.

Ожидается, что пользователь может пройти сценарий без ручного ввода callback data.

### Сценарий 3. Углубленная траектория

Проверка:

1. Отправить 1-5 ссылок на программы `51.pfdo.ru`.
2. Проверить обзор пройденных программ и классифицированных тем.
3. Нажать `Показать все темы`, если кнопка доступна.
4. Если программы из разных муниципалитетов, выбрать муниципалитет.
5. Проверить ввод другого населенного пункта вручную.
6. Если возраст не выведен, ввести возраст числом.
7. Изменить критерии или пропустить изменение.
8. Получить список следующих программ.
9. Скачать PDF.

Ожидается:

- невалидные ссылки не ломают сценарий;
- отсутствующие program id перечисляются пользователю;
- если карточки нет в зеркале, on-demand импорт не ломает сценарий;
- уже пройденные программы не попадают в результат;
- объяснение содержит продолженные темы и признаки углубления.

### Сценарий 4. Траектория новых интересов

Проверка:

1. Запустить сценарий кнопкой `Траектория новых интересов` или командой `/wide`.
2. Отправить 1-5 ссылок на программы `51.pfdo.ru`.
3. Проверить обзор пройденных программ.
4. Выбрать населенный пункт и при необходимости изменить критерии.
5. Получить список программ по новым направлениям.
6. Скачать PDF.

Ожидается:

- результат объясняет новые темы и отличие от уже изученного;
- пройденные программы не попадают в выдачу;
- программы с повторяющейся level 2 категорией не должны вытеснять новые направления;
- если тем нет, бот просит пользователя описать интересы вручную.

### Web-chat

Проверьте:

- `POST /web-chat/message`;
- `POST /web-chat/callback`;
- `POST /web-chat/reset`;
- скачивание `GET /web-chat/document/<token>`.

Проверьте CORS с реального домена сайта и с localhost, если это нужно для разработки.

На landing page дополнительно проверьте:

- кнопки меню web-chat `/text`, `/quiz`, `/deep`, `/wide`, `/help`;
- footer-ссылки Telegram, MAX и Mattermost;
- наличие счетчика Яндекс Метрики в `<head>`;
- отсутствие JS-ошибок при открытии и отправке сообщения.

### Mattermost

Проверьте:

- личное сообщение боту;
- сообщение в канале с mention;
- выбор варианта по номеру;
- мультиселект через несколько номеров, разделенных запятой;
- ответы в thread, если включен `MATTERMOST_REPLY_MODE=thread`.

### MAX

Проверьте:

- стартовое событие;
- обычное сообщение;
- callback;
- командное меню `/text`, `/quiz`, `/deep`, `/wide`, `/help`;
- отправку PDF.

### Алиса

Алиса пока не проходит полный подбор. Для приемки достаточно проверить:

- стартовый ответ;
- help-запрос;
- ping, если он используется в настройках навыка.

## Проверка PFDO pipeline

После полного обновления данных:

```bash
psql -d pfdo_51_mirror -c "select count(*) from pfdo_programs;"
psql -d pfdo_51_mirror -c "select count(*) from pfdo_program_calendar_topics;"
psql -d pfdo_51_mirror -c "select count(*) from pfdo_program_topic_aggregates;"
node scripts/evaluate-pfdo-topic-classifier.js
```

Проверьте:

- количество программ не стало неожиданно нулевым;
- документы скачиваются;
- темы извлекаются;
- `pfdo_sync_runs` показывает успешный последний запуск;
- `pfdo_program_sync_state` отражает статусы документов и тем;
- очередь ручной проверки не выросла резко без причины;
- углубленная траектория продолжает находить кандидатов;
- траектория новых интересов не возвращает простые повторы уже изученной категории.

## Проверка parser auto-updater

Dry-run:

```bash
node scripts/update-pfdo-program-parser.js \
  --program-ids exports/parser_update_programs.csv \
  --limit 1
```

Проверить:

- отчет в `exports/parser-updater-report.csv`;
- артефакты в `tmp/parser-updater/<program_id>/`;
- статус `ok`, `needs_update`, `needs_reload_local` или другой ожидаемый статус;
- отсутствие неожиданных изменений в коде.

Apply-режим разрешен только после dry-run и просмотра артефактов.

## Критерии приемки изменений

### Изменение пользовательского сценария

Готово, если:

- сценарий описан в `docs/scenario-*.md` или связанном документе;
- тесты сценария проходят;
- ручной happy path проходит;
- неполный ввод обрабатывается понятно;
- устаревшие callback-и не запускают неактивный сценарий;
- история рекомендаций сохраняется.

### Изменение рекомендаций

Готово, если:

- обновлен [recommendation-engine.md](recommendation-engine.md);
- strict-режим не возвращает mock-данные;
- результат не превышает запрошенный limit;
- карточка содержит ссылку PFDO;
- неизвестное расписание объясняется пользователю;
- ручные кейсы не стали очевидно хуже.

### Изменение PFDO pipeline

Готово, если:

- обновлена [pfdo-data-pipeline.md](pfdo-data-pipeline.md);
- схема и документация схемы синхронизированы;
- есть проверка на ограниченном наборе программ;
- regression-тесты парсера проходят;
- есть понятный rollback или snapshot для изменяемых строк.

### Изменение конфигурации или деплоя

Готово, если:

- обновлены `.env.example` и [configuration-reference.md](configuration-reference.md);
- обновлен [operator-guide.md](operator-guide.md), если меняются команды сопровождения;
- `/health` проходит после деплоя;
- секреты не попали в репозиторий.

## Что считать блокером релиза

- `npm test` падает.
- `/health` не отвечает.
- Бот не отвечает на `/start` в основном канале.
- `DATABASE_URL` или `PFDO_MIRROR_DATABASE_URL` указывают не туда.
- Быстрый сценарий в strict-режиме показывает mock-данные.
- PDF не создается в канале, где он заявлен как поддерживаемый.
- После обновления PFDO-зеркала `pfdo_programs` пустая.
- Изменения требуют секретов в репозитории.

## Что фиксировать после приемки

После каждого релиза полезно сохранить:

- дату и состав релиза;
- результат `npm test`;
- результат `/health`;
- список проверенных каналов;
- известные ограничения;
- ссылку на commit или PR;
- список миграций или PFDO-операций, если они выполнялись.

Эти записи можно вести в будущем `docs/release-notes.md` или в PR-описании.
