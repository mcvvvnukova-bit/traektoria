# Безопасность и приватность

Статус: первый рабочий черновик от 2026-06-13.

Документ описывает основные риски безопасности проекта и правила работы с пользовательскими данными, секретами, логами и артефактами.

## Какие данные обрабатывает бот

В диалогах могут появляться:

- возраст ребенка;
- интересы ребенка;
- населенный пункт или район;
- удобное расписание;
- бюджет семьи;
- особенности ребенка;
- ограничения по формату занятий;
- ссылки на пройденные программы;
- id чата или клиента в канале.
- username пользователя в канале, если платформа его передала.

Это не медицинская система и не система записи. Но часть данных может быть чувствительной, особенно сведения об особенностях ребенка.

## Где хранятся данные

| Данные | Где хранятся |
| --- | --- |
| Состояние диалога | `telegram_bot.bot_sessions` |
| Runtime state | `telegram_bot.bot_runtime_state` |
| История рекомендаций | `telegram_bot.recommendation_history` |
| Каталог PFDO | `pfdo_51_mirror` |
| Документы программ | `tmp/program_docs` |
| Parser updater artifacts | `tmp/parser-updater` |
| Экспорты аналитики | `exports/` |
| PDF-подборки | `PDF_OUTPUT_DIR` или временный каталог ОС |

В `bot_sessions.metadata` и `recommendation_history.metadata` хранится ограниченная транспортная metadata: стабильный `userId`, если он доступен, и `username`, если он есть у пользователя. `username` может отсутствовать или измениться, поэтому он не используется как ключ и может быть сохранен как `null`.

`tmp/` и `exports/` игнорируются через `.gitignore`.

## Секреты

Нельзя коммитить:

- `.env`;
- `.env.*`, кроме `.env.example`;
- токены Telegram и MAX;
- пароль Mattermost;
- personal access token Mattermost;
- `OPENAI_API_KEY`;
- строки подключения к приватным БД с паролями;
- webhook secret paths;
- приватные ключи и сертификаты.

`.gitignore` уже исключает `.env`, ключи, сертификаты, `tmp/`, `exports/`, логи и кеши.

## Webhook security

### Telegram

Используйте:

- случайный `TELEGRAM_WEBHOOK_PATH`;
- `TELEGRAM_WEBHOOK_SECRET_TOKEN`;
- HTTPS на публичном домене;
- nginx proxy только на нужные пути.

Код проверяет заголовок `x-telegram-bot-api-secret-token`, если задан `TELEGRAM_WEBHOOK_SECRET_TOKEN`.

### MAX

Используйте:

- случайный `MAX_WEBHOOK_PATH`;
- `MAX_WEBHOOK_SECRET`;
- HTTPS.

Код проверяет заголовок `x-max-bot-api-secret`, если задан `MAX_WEBHOOK_SECRET`.

### Алиса

Используйте секретный `ALICE_WEBHOOK_PATH`. Полный сценарий подбора через Алису пока не включен, но endpoint все равно должен быть непредсказуемым.

## Web-chat

Web-chat защищается:

- `WEB_CHAT_ALLOWED_ORIGINS`;
- `WEB_CHAT_BODY_LIMIT_BYTES`;
- `WEB_CHAT_MESSAGE_MAX_LENGTH`;
- `WEB_CHAT_RATE_LIMIT_PER_MINUTE`;
- временными ссылками на PDF через `WEB_CHAT_DOCUMENT_TTL_MS`.

Проверяйте список origins при каждом изменении домена сайта. Не ставьте wildcard без отдельного решения.

## Mattermost

Предпочитайте token-based авторизацию через `MATTERMOST_TOKEN`, если это принято в инфраструктуре.

Если используются логин и пароль:

- храните их только в `.env` или менеджере секретов;
- не вставляйте пароль в документацию;
- ограничьте права аккаунта бота;
- проверьте, что бот не отвечает на собственные сообщения.

Для каналов используйте режим `MATTERMOST_MODE=mentions`, если бот не должен реагировать на все сообщения.

## PostgreSQL

Рекомендации:

- используйте отдельные базы `telegram_bot` и `pfdo_51_mirror`;
- ограничивайте доступ пользователя БД нужными правами;
- не используйте суперпользователя PostgreSQL для runtime без необходимости;
- делайте backup перед изменениями схемы и массовыми refresh-операциями;
- не публикуйте дампы с пользовательскими сессиями.

Особенно чувствительна таблица `recommendation_history`, потому что она хранит payload запросов и ответов, а также transport metadata пользователя.

## Логи

В логах могут оказаться:

- ошибки транспорта;
- ошибки БД;
- ошибки PFDO;
- тексты исключений;
- части пользовательского контекста при отладке.

Правила:

- не логировать токены и пароли;
- не добавлять в обычные логи полный `.env`;
- не публиковать `journalctl`-вывод без просмотра;
- чистить или маскировать пользовательские данные при передаче логов вне команды.

## PDF и временные файлы

PDF-файлы могут содержать:

- возраст;
- место;
- интересы;
- ограничения;
- список рекомендованных программ.

Рекомендации:

- хранить PDF во временном каталоге или ограниченном `PDF_OUTPUT_DIR`;
- не индексировать каталог с PDF;
- не публиковать прямые файловые пути;
- для web-chat полагаться на временные token URL;
- периодически чистить старые PDF, если каталог не временный.

## OpenAI и локальная LLM

Пользовательский runtime использует локальную LLM только при `LOCAL_LLM_ENABLED=true`. Она должна быть OpenAI-compatible и работать по `LOCAL_LLM_API_URL`.

OpenAI API используется parser auto-updater-ом для аудита документов и repair plan. Это сервисный процесс, не основной пользовательский диалог.

Перед запуском parser auto-updater с OpenAI:

- проверьте, какие фрагменты документов попадут в запрос;
- не отправляйте секреты;
- используйте кеш только в локальном защищенном каталоге;
- не коммитьте `tmp/parser-updater-cache`.

## Детские данные и особенности ребенка

Бот может получать сведения об ОВЗ, РАС, СДВГ, зрении, слухе и других особенностях. Эти данные нужны только для подбора подходящего формата занятий.

Правила:

- не использовать такие данные для публичной аналитики без обезличивания;
- не выводить их в логи без необходимости;
- не передавать третьим лицам без правового основания;
- не делать медицинских выводов;
- формулировать рекомендации как помощь в выборе формата, а не диагноз.

## Минимизация данных

Для развития проекта стоит рассмотреть:

- TTL или регламент очистки `bot_sessions`;
- отдельную политику хранения `recommendation_history`;
- маскирование chat id, user id и username в аналитических выгрузках;
- удаление старых PDF;
- отдельные роли БД для runtime и maintenance-скриптов.

## Проверка перед публикацией артефактов

Перед отправкой логов, CSV, JSON, PDF или скриншотов вне команды проверьте:

1. Нет токенов и паролей.
2. Нет приватных строк подключения.
3. Нет webhook secret paths.
4. Нет персональных chat id, user id или username без необходимости.
5. Нет чувствительных сведений о ребенке.
6. Нет скачанных документов программ, если их нельзя распространять.

## Incident response

Если секрет попал в репозиторий или внешний канал:

1. Немедленно отзовите или перевыпустите секрет.
2. Обновите `.env` на сервере.
3. Перезапустите сервис.
4. Проверьте логи на несанкционированные запросы.
5. Удалите секрет из истории публикации, если это возможно.
6. Зафиксируйте инцидент и меры.

Если утекли пользовательские данные:

1. Определите объем данных.
2. Ограничьте дальнейший доступ.
3. Уведомите ответственных за проект.
4. Сохраните факты без распространения лишних копий.
5. Согласуйте дальнейшие действия с владельцем данных.
